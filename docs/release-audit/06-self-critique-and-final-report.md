# Mandatory adversarial self-critique and final report

Before any final result, answer in writing:

- Which instruction was easiest to overlook?
- Did work focus only on the latest complaint while older requirements remained incomplete?
- Was a concept image mistaken for approved implementation scope?
- Are any requirements represented only by test names?
- Were useful regressions removed, thresholds lowered or validators weakened?
- Does the installed app visibly differ from the old version?
- Do any old build numbers, labels, duplicate controls or inactive buttons remain?
- Is the top staff unobstructed in every required layout?
- Does mouse docking work exactly as promised?
- Does menu grouping match the approved design?
- Was every visible control used?
- Were PDF and every PNG opened and inspected?
- Were startup and editing performance measured?
- Was installer upgrade identity verified by path and hash?
- Which claims rely only on source, only on CI, or still require a person/hardware?
- Would a skeptical reviewer accept every PASS?

Any discovered defect returns work to the relevant earlier checkpoint.

The final report must record repository, branch, version, build, release commit/message, workflow/run/conclusion, exact tests, artifact names/sizes/hashes, process/install paths, About version, badge, requirement totals by status, staff-bar result, docking result, menu result, old-UI removal, publishing action results, PDF/PNG inspection, performance measurements, file association, lyric/data regressions, untested items, self-critique and unresolved risks.

Release decision must be exactly one:
- APPROVED FOR DOWNLOAD
- RELEASE BLOCKED
